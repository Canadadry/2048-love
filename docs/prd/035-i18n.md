# Internationalisation (i18n)

## Problem Statement

All user-facing strings (screen titles, menu labels, option names, HUD text) are hardcoded in English across the screen and HUD modules. There is no mechanism for players to play the game in their preferred language.

## Solution

Introduce a CSV-driven translation table (`assets/i18n.csv`) with one column per supported language. A new `lib/i18n` module parses the CSV at startup and resolves any symbolic key to the active language's string, falling back to English then the raw key. A new Language row in the Options screen lets players pick their language; the choice persists across launches via the settings system. All hardcoded strings in screens and the HUD are replaced with `i18n.t("symbolic.key")` calls. The initial CSV ships with English and French.

## User Stories

1. As a player, I want to choose my preferred language from the Options screen, so that all menus and labels appear in that language.
2. As a player, I want my language choice to be remembered across launches, so that I don't have to re-select it every time I open the game.
3. As a player, I want the language option to show native language names (e.g. "Français" not "French"), so that I can identify my language regardless of the current active language.
4. As a player, I want navigating left/right on the Language row to cycle through available languages immediately, so that the change is as responsive as other options.
5. As a player, I want the main menu labels ("New Game", "Options", "Quit") displayed in my selected language.
6. As a player, I want the pause screen labels ("Resume", "New Game", "Main Menu", "Quit", "Paused") displayed in my selected language.
7. As a player, I want the win screen labels ("You Win!", "Continue", "Restart", "Main Menu") displayed in my selected language.
8. As a player, I want the game over screen labels ("Game Over", "New Game", "Main Menu") displayed in my selected language.
9. As a player, I want the options screen labels ("Options", "Win Tile", "Theme", "Animations", "Effects", "Sound", "Language", "Back", "ON", "OFF") displayed in my selected language.
10. As a player, I want the HUD score prefix ("Score: ") displayed in my selected language, including correct typographic conventions (e.g. French space before colon).
11. As a player, if a translation is missing for my language, I still see the English string rather than a broken key.
12. As a developer, I want to add a new language by adding a column to the CSV with no code changes, so that the system is open for extension.
13. As a developer, I want the i18n module to be testable in isolation without a Love2D runtime, by accepting CSV content as a string rather than reading from disk itself.
14. As a developer, I want the default language and the CSV file path to be defined in `config`, so that they are easy to find and change without touching the i18n module.
15. As a developer, I want `lib/i18n` to have no dependency on `config` or `settings`, keeping it a clean, reusable module.

## Implementation Decisions

- **`config.lua`** gains two constants: `LANG_FILE` (path to the CSV, relative to the Love2D source directory) and `DEFAULT_LANG` (the language code used on first launch, `"en"`).

- **`game/assets/i18n.csv`** is the translation table. Format: first row is a header (`key,en,fr,...`); each subsequent row is `symbolic.key,value_en,value_fr,...`. The language codes in the header become the canonical identifiers. The special key `lang.name` holds the native display name for each language (e.g. `English`, `Français`). Initial languages: `en`, `fr`.

  The initial key set:
  `lang.name`, `screen.paused`, `screen.game_over`, `screen.options`, `screen.you_win`,
  `menu.new_game`, `menu.quit`, `menu.resume`, `menu.main_menu`, `menu.back`,
  `menu.continue`, `menu.restart`,
  `options.win_tile`, `options.theme`, `options.animations`, `options.effects`,
  `options.sound`, `options.language`, `options.on`, `options.off`,
  `hud.score`.

- **`game/lib/i18n.lua`** (new deep module):
  - `i18n.load(csv_string)` — parses the CSV string, builds an internal lookup table `{key → {lang_code → value}}`. Determines the ordered language list from the header row. Resets any previously loaded data. No I/O.
  - `i18n.set_lang(code)` — sets the active language. If `code` is not in the loaded language list, defaults to `"en"`.
  - `i18n.lang()` — returns the current active language code.
  - `i18n.languages()` — returns an ordered list of `{code, name}` pairs, where `name` is the value of the `lang.name` key for that language. Used by the Options screen to populate the Language row.
  - `i18n.t(key)` — returns the translation for `key` in the current language. Fallback chain: current language → `"en"` → the raw key string itself.
  - No dependency on `config`, `settings`, or `love.*`.

- **`game/main.lua`** initialises i18n at startup, after settings are loaded:
  ```lua
  local csv = love.filesystem.read(config.LANG_FILE)
  i18n.load(csv)
  i18n.set_lang(settings.get("language", config.DEFAULT_LANG))
  ```

- **`game/screens/options_screen.lua`** adds a Language row using the same `value_row` helper as existing rows. The value list comes from `i18n.languages()`, the getter is `i18n.lang()`, the setter calls `i18n.set_lang(v)` and `settings.set("language", v)`, and the format function returns the `name` field of the `{code, name}` pair. Changing language takes effect immediately for all subsequent draws.

- **All hardcoded strings in `game/screens/*.lua` and `game/hud.lua`** are replaced with `i18n.t("symbolic.key")` calls. The `"None (classic)"` theme label in `options_screen.lua` is replaced with `i18n.t("options.theme_none")` and added to the CSV.

- **Language persistence**: the `settings` key is `"language"`, storing the language code string. Consistent with how `animations_enabled`, `sound_enabled`, and `theme` are persisted today.

- **Language switching at runtime**: since all `spec()` / `draw()` calls re-evaluate `i18n.t()` each frame, switching language is immediately visible on the next draw cycle with no additional wiring.

## Testing Decisions

- Tests assert observable behavior through the public interface, not internal table structure — consistent with existing test style across `game/lib/*_test.lua`.
- **`game/lib/i18n_test.lua`** (new) covers:
  - `t()` returns the correct value for the active language.
  - `t()` falls back to English when the current language has no value for a key.
  - `t()` falls back to the raw key when the key is absent from both current language and English.
  - `set_lang()` with an unknown code silently falls back to `"en"`.
  - `languages()` returns the correct ordered list of `{code, name}` pairs derived from the CSV header and `lang.name` row.
  - `load()` called a second time resets state from the first load.
  - All tests inject CSV content as a string directly into `i18n.load()` — no file I/O, no Love2D dependency.
- Prior art: `game/lib/menu_screen_test.lua`, `game/lib/optionsmodel.lua` tests — pure-Lua module tests with inline fixture data.

## Out of Scope

- OS locale auto-detection — the default language is always `config.DEFAULT_LANG` (`"en"`); the player picks their language via Options.
- Right-to-left layout or font changes for RTL languages.
- Pluralisation rules or format strings with variable interpolation.
- Translation of theme names sourced from tileset filenames (only the "None (classic)" fallback label is translated).
- A translation workflow or tooling for contributors — the CSV is the sole translation artefact.
- Languages beyond `en` and `fr` at launch — adding more requires only a new CSV column.

## Further Notes

- The `lib/i18n` module is intentionally a deep module: a small, stable interface hides all CSV parsing, language-list management, and fallback logic. It can be tested in full isolation and never needs to know how the CSV arrived or where the language choice is stored.
- Language names are stored as the `lang.name` key in the same CSV, avoiding a separate config file or hardcoded map.
- French typography requires a non-breaking space before `:` — the `hud.score` key value for `fr` should be `"Score\194\160: "` (UTF-8 non-breaking space U+00A0) to respect this convention.
